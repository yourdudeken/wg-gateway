# Contributing to wg-gateway

First off, thank you for considering contributing to `wg-gateway`! It's people like you that make the open-source community such an amazing place to learn, inspire, and create.

## Code of Conduct

By participating in this project, you agree to abide by our Code of Conduct. (Policy: Be excellent to each other).

## How Can I Contribute?

### Reporting Bugs
* Check the existing issues to see if the bug has already been reported.
* If not, open a new issue. Clearly describe the problem, including steps to reproduce it.
* Include details about your environment (OS, Go version, Docker version).

### Suggesting Enhancements
* Open a new issue and describe the feature you'd like to see.
* Explain why this feature would be useful to others.

### Pull Requests
1. Fork the repository.
2. Create a new branch for your feature or fix: `git checkout -b feature/my-new-feature` or `git checkout -b fix/my-fix`.
3. Commit your changes with descriptive messages: `git commit -am 'Add some feature'`.
4. Push to the branch: `git push origin feature/my-new-feature`.
5. Submit a pull request.

## Development Setup

### Prerequisites
- **Go 1.18+**
- **Docker** and **Docker Compose** (for testing deployments)

### Building
```bash
go build -o wg-gateway main.go
```

### Running Tests
```bash
go test ./...
```

## Project Structure
- `cmd/`: CLI commands using Cobra.
- `internal/config/`: Configuration handling (YAML).
- `internal/templates/`: Embedded templates and rendering logic.
- `internal/wg/`: WireGuard specific logic (key generation).
- `deploy/`: (Ignore) This is generated by the tool.

## Style Guide
- Follow standard Go formatting (`go fmt`).
- Use descriptive variable and function names.
- Keep functions small and focused.
- Document public APIs.

## Questions?
Feel free to open an issue with the "question" label.
